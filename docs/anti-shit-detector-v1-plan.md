# 反屎文检测器 V1 方案

## 目标

做一个面向小红书家居 / 生活类内容的“反屎文检测器”。

它不是教用户怎么写爆款，而是先指出：

- 文章为什么烂
- 烂在哪一段
- 它属于哪一种常见烂法
- 最优先该删什么、补什么、改什么

V1 目标不是追求全平台、全模态，而是先把“坏内容诊断”做得尖锐、具体、可执行。

## 产品定位

一句话版本：

> 帮你判断这篇内容是不是空、装、假、模板、标题诈骗，并告诉你怎么止损。

和常见“爆款生成器”的差异：

- 生成器偏正向创作
- 反屎文工具偏负向体检
- 生成器强调“怎么写更好”
- 反屎文工具强调“现在到底烂在哪里”

建议外部名称避免过度攻击性。内部可叫“反屎文检测器”，外部可用：

- 烂文检测器
- 反套路笔记体检
- 废话笔记扫描仪
- 内容止损诊断器

## V1 范围

只做：

- 类目：`home` / `lifestyle`
- 输入：标题、正文、1-9 张图片
- 输出：
  - 烂文指数
  - 维度分
  - 烂点标签
  - 逐段点评
  - 犀利总结
  - 可执行修改建议
  - 一版“止损改写”

暂不做：

- 视频
- 全平台
- 自动大规模在线抓取
- 用户账号画像
- 长链路历史追踪

## 为什么先做家居 / 生活

这两个类目里最常见的低质量内容非常集中：

- 假治愈
- 假松弛
- 伪经验分享
- 图好文烂
- 模板拼贴
- 情绪堆砌但没信息

这意味着 taxonomy 更容易定义，模型输出也更稳定。

## 输入输出设计

### 输入

- 标题
- 正文
- 图片数量与图片本身
- 可选标签
- 可选发布链接或截图（后续版本）

### 输出

- `trash_index`: 0-100，越高越烂
- `dimension_scores`
  - 空洞度
  - 模板味
  - 标题诈骗度
  - 信息密度不足
  - 图文割裂度
  - AI 腔浓度
- `failure_tags`
- `top_issues`
- `paragraph_breakdown`
- `sharp_summary`
- `salvage_actions`
- `rewrite_direction`

## “坏文章”定义原则

V1 不用“主观审美差”做核心判断，而用可解释的失败模式：

1. 承诺与兑现不匹配
2. 有情绪，没有信息
3. 有结构，没有内容
4. 有观点，没有证据 / 场景 / 细节
5. 图片在撑场面，文字没完成说明任务
6. 语言像模板或 AI 口水

## V1 taxonomy

建议先做 8 类：

1. 假干货型
2. 假治愈型
3. 假松弛型
4. 空方法论型
5. 模板拼贴型
6. 标题诈骗型
7. 图好文烂型
8. AI 口水型

每一类都需要：

- 定义
- 典型特征
- 常见口头禅 / 套话
- 判定信号
- 纠正方式

## 评分维度

### 1. 空洞度
判断是否只有态度和形容词，没有具体信息。

### 2. 模板味
判断是否像拼接爆款模板，缺乏真实细节与个体经验。

### 3. 标题诈骗度
判断标题承诺的收益、数量、反差，是否在正文兑现。

### 4. 信息密度不足
判断是否缺数字、步骤、材料、约束条件、结果对比。

### 5. 图文割裂度
判断图片承担的叙事是否和文字对应，还是纯氛围图撑场。

### 6. AI 腔浓度
判断是否存在过度平滑、泛化、无对象感、无生活摩擦的文字。

## 数据研究方案

### 研究样本来源

先用 Scrapling 做小规模研究样本，不做大规模长期抓取。

关键词建议：

- 家具
- 家居
- 装修
- 收纳
- 独居
- 生活方式
- 小家改造
- 租房改造
- 软装
- 氛围感

### 初始筛选条件

- 图片数 >= 1
- 正文字数 >= 120
- 正文字数 <= 1200
- 公开内容
- 排除纯商品卡 / 纯空壳转发 / 无正文帖子

### 样本规模建议

研究 V1：

- 抓取 200-500 条样本
- 人工精读 50 条
- 形成第一版 taxonomy

不要一上来追求几千条，因为第一阶段重点是“定义坏”，不是堆量。

### 坏样本筛选方法

不要直接用低互动 = 坏内容。

建议三层混合：

1. 弱信号：互动表现差
2. 结构信号：文本与图片特征差
3. LLM 初筛：判断失败模式

最后再做少量人工校准。

## 模型与系统分工

### 规则层

负责稳定信号：

- 标题长度
- 标题承诺词
- 夸张词密度
- 数字 / 材料 / 步骤 / 场景词比例
- 段落结构
- 重复句式
- 标点和 emoji 滥用

### 视觉层

负责基础图文一致性判断：

- 图片数量
- 场景一致性
- 是否全是氛围图
- 是否能支撑标题承诺

### 主诊断模型

负责输出结构化诊断：

- failure tags
- sharp issues
- paragraph breakdown
- salvage actions

### Grok

只负责表达风格：

- 犀利总结
- 毒舌转述
- 更有攻击性的文案包装

不建议让 Grok 负责唯一判定逻辑。

## 输出语气原则

产品可以尖锐，但不要变成单纯羞辱。

正确目标：

- 准
- 狠
- 可改

错误目标：

- 单纯骂
- 情绪大于诊断
- 只有评价，没有修法

## 风险

### 抓取合规风险

- 平台条款
- 反爬
- 请求频率
- 内容存储与再利用

建议：

- 小样本研究优先
- 缓存
- 节流
- 不抓敏感信息
- 不宣传“大规模平台采集”

### 定义漂移风险

“坏”容易越做越主观。

建议：

- taxonomy 先于模型自由发挥
- 所有类目先有正反例
- 保留人工校准闭环

### 表达越界风险

建议保留两种输出风格：

- 默认：专业但狠
- 开关：毒舌版

## 开发阶段

### Phase 1：研究与定义

- 建研究样本
- 建 taxonomy
- 建评分维度
- 产出坏文信号字典

### Phase 2：本地 demo

- 用户输入标题 / 正文 / 图片
- 前端跑基础规则
- 后端跑结构化诊断
- 输出报告与修改建议

### Phase 3：风格增强

- 接 Grok 做犀利点评层
- 让输出更有传播性

## 2 周 V1 路线

### 第 1 周

- 完成 taxonomy
- 完成样本抓取与筛选
- 做规则引擎第一版
- 做结构化 report schema

### 第 2 周

- 接一个可用的诊断模型
- 做输入 / 输出页面
- 接图片分析
- 做 demo 版 sharp summary
